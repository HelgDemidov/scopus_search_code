## Обновление статуса (после 2026-07-03)

Ниже оценки по-прежнему актуальны — коротко, что добавилось с момента написания

## Общая оценка

Инженерно это заметно выше типичного «портфолио»-проекта — но как демонстрационный
кейс для работодателя в середине 2026-го ему не хватает одного явного акцента:
применения LLM/AI к собственному домену. Ниже — по пунктам, без реверансов.

## Что реально сильно (не для галочки)

**Backend-дисциплина выше среднего для соло-проекта:**
- Слоистая архитектура interfaces→services→infrastructure→routers выдержана
  последовательно, не декларативно — это редкость, обычно к 3-му месяцу проекта
  слои размываются.
- Auth сделан по-взрослому: AT только in-memory, RT-ротация с piggyback-очисткой
  стухших токенов при каждом рефреше, password reset с revoke-all-tokens после
  смены пароля. Это не «два месяца туториала», а реальное понимание threat model.
- Найденный и починенный баг с cache namespace (prod/staging делили один
  Redis-ключ) — отличная «военная история» для собеседования: обнаружение →
  root cause → фикс → регрессионный тест. Именно такие кейсы, а не список
  технологий, продают senior-мышление.
- CI-матрица (unit/PG16/alembic drift check/coverage gate/lint/e2e на живом
  staging) — это уровень небольшой продуктовой команды, не пет-проекта.

**Сегодняшняя работа (Table Builder + Journal Landscape Scatter) — качественная
инженерия:**
- Whitelist из C(5,2)=10 пар измерений с server-side валидацией + client-side
  логикой «просто исключить выбранное» без дублирования whitelist — элегантное
  решение, видно, что думали о инвариантах, а не просто «сделали, чтобы работало».
- Отдельная забота о non-cached vs cached эндпоинтах, marginal totals vs grand
  total в pivot — не очевидные UX-ловушки, которые вы явно предвидели и объяснили
  пользователю в UI (сноска про totals).
- Scatter-чарт с квадрантами и halo-эффектами — это единственный по-настоящему
  запоминающийся визуальный элемент во всём /explore. Остальное — компетентный,
  но стандартный SaaS-dashboard набор (KPI-тайлы, bar/donut/line).

## Где не дотягивает до стандартов июля 2026

**1. Отсутствие AI-слоя — самый большой пробел.** У вас на руках корпус из
122k статей с abstract'ами, а поиск — чисто CQL/keyword. В середине 2026 для
домена «поиск и анализ научных публикаций» отсутствие семантического поиска
(embeddings + vector search) или хотя бы одной LLM-фичи (авто-саммари тренда,
natural-language → pivot-запрос типа «покажи open access тренд по Германии
с 2020») читается работодателем как «не трогал этот стек руками». Это не
украшение ради моды — это ровно тот кейс, где RAG действительно уместен по
сути задачи, а не притянут искусственно.

**2. Инженерные истории не вынесены наружу.** Баг с cache namespace,
whitelist-паттерн, RT cleanup piggyback — всё это живёт в CLAUDE.md и коммитах,
которые рекрутер никогда не откроет. Если в README нет секции вида
«Engineering decisions / Lessons learned» с 3-5 такими кейсами в формате
«проблема → решение → чему научился» — вы теряете 80% ценности уже проделанной
работы просто из-за упаковки.

**3. Observability на уровне «Railway логи», а не приложения.** Нет
структурированного логирования, трейсинга (OpenTelemetry) или Sentry.
Учитывая, что CI-дисциплина уже на высоком уровне, это дешёвое дополнение
(несколько часов) с непропорционально большим сигналом «понимает production».

**4. Нет истории производительности/нагрузки.** Ни одного числа вида «P95
латентность X мс на Y RPS» или синтетического нагрузочного теста. При корпусе
в 122k статей это не про масштаб, а про то, что вы ни разу не измерили —
а измерить и написать 2 абзаца в README дёшево.

**5. A11y проверяется вручную, не автоматически.** Атрибуты (aria-sort, role,
фокус-кольца) поставлены осознанно — это плюс. Но нет lighthouse-CI/axe-core
джобы, которая бы это закрепляла — учитывая, что автоматизировано уже всё
остальное (lint/coverage/alembic drift), это единственный «ручной» угол QA.

**6. Визуальная идентичность — компетентно, но не отличимо.** Navy/slate/blue
KPI-дашборд с donut/bar/line — таких тысячи на Dribbble и в туториалах shadcn.
Scatter с квадрантами — ваш единственный по-настоящему авторский элемент; стоит
инвестировать ещё в 1-2 таких же нестандартных визуализации, а не в шестой bar
chart.

## Приоритизация, если время ограничено

1. **README как case study** — бесплатно с точки зрения кода, самый высокий ROI
   для работодателя.
2. **Одна AI-фича** — даже простая (embeddings для semantic search по
   названиям/abstract'ам через pgvector, или LLM-суммари выбранного pivot-среза)
   закрывает самый заметный пробел.
3. **Sentry/OTel** — несколько часов, сильный сигнал зрелости.
4. Остальное (нагрузочное тестирование, a11y-автоматизация, доп. визуализации) —
   по вкусу, уже не критично.

## Реализация приоритетных задач

### Observability — без внешних сервисов (2026-07-06)

Первая версия плана предполагала Sentry (backend + frontend). При разборе
всплыли реальные развилки, а не просто детали реализации: Sentry требует
создания SaaS-аккаунта (внешнее действие, которое нельзя сделать не глядя),
решения — один Sentry-проект на prod/staging с тегом `environment` или два
раздельных (прямая параллель с прошлым багом namespace-коллизии в
Redis-кэше, PR #32 — то же самое «прод и staging делят один ресурс»), явной
PII-политики (`send_default_pii`), а для фронтенда — отдельного CI-шага с
source maps, без которого стектрейсы бесполезны.

Решение: закрыть пробел без единого нового внешнего сервиса — это снимает
все три развилки разом и не создаёт новых аккаунтов/секретов/окружений.

**Что делаем (~3-4ч):**
1. Structured JSON logging (`structlog`) + middleware с `request_id`
   (contextvars) → тот же id в заголовке ответа `X-Request-ID`.
2. Global exception handler — необработанные исключения логируются как
   structured JSON (ERROR) с полным traceback + request-контекстом.
3. `ping()` в `UpstashRedisClient` (`app/infrastructure/redis_client.py`,
   сейчас только `get`/`setex`) + эндпоинт `/health/redis` — отличает
   «процесс жив» от «процесс жив, но Redis лежит» (`/health/db` уже есть).
4. Алертинг piggyback на существующий 2-часовой seeder-cron: в `seed_db.py`,
   рядом с вызовом `/seeder/gc`, проверка `/health`, `/health/db`,
   `/health/redis`; при не-200 — письмо через уже существующий
   `BrevoEmailService` (`IEmailService`).
5. Разовый расчёт p95 latency из structured-логов за сутки → 2 абзаца в
   README (заодно частично закрывает п.4 «нет истории производительности»).

**Осознанно не делаем** (не полдела, а явный trade-off, стоит называть в
README как есть, а не подавать как «observability полностью закрыта»):
дедупликация/группировка ошибок (одна exception = N строк лога, не «1 issue,
N раз»), задержка алертинга до 2ч (привязана к циклу cron, не realtime),
видимость ошибок фронтенда (self-built версия — по сути переизобретение
куска Sentry), retention логов ограничен окном самого Railway.

Тикет: issue #48.

**Статус (2026-07-06): реализовано**, ветка `feat/observability` смёрджена в
`main` напрямую (без PR — CI на ветке уже был зелёным), коммиты `ad00dd7` +
merge `241302f`.

Пункты 1–4 из плана выше закрыты полностью:
- `structlog` (JSON) + `RequestIDMiddleware` (`app/core/logging_config.py`) —
  `request_id` через contextvars, тот же id и в заголовке ответа
  `X-Request-ID`, и в каждой строке лога этого запроса.
- Global exception handler (`app/main.py`) — необработанные исключения
  логируются как structured ERROR с полным traceback; клиенту уходит только
  generic `{"detail": "Internal server error"}`, без утечки деталей.
- `UpstashRedisClient.ping()` + `GET /health/redis` (дополняет уже
  существовавшие `/health` и `/health/db`) — различает «процесс жив» и
  «процесс жив, но Redis лежит».
- `POST /seeder/health-check` — piggyback на seeder cron (тот же
  `X-Seeder-Secret`, что и `/seeder/gc`): проверяет БД (`SELECT 1`) и Redis
  (`ping()`) in-process, при деградации шлёт письмо через новый метод
  `BrevoEmailService.send_alert_email()` (добавлен в `IEmailService`).
- Тесты: `tests/integration/test_health.py` (5 шт.), `test_observability.py`
  (3 шт.), плюс 5 новых в `test_seeder_endpoint.py` — все на SQLite, без
  `requires_pg`.
- Побочная находка и фикс по ходу: у 500-ответов не проставлялся
  `X-Request-ID` — голый `Exception`-handler у Starlette выполняется в
  `ServerErrorMiddleware`, который стоит выше `RequestIDMiddleware` в стеке
  мидлварей, поэтому код после `call_next()` в мидлваре для этого пути не
  успевает отработать; `request_id` теперь читается напрямую из
  `structlog.contextvars` внутри самого exception handler'а.

Пункт 5 (расчёт p95 latency в README) не реализован — требует накопленной
live-статистики с продакшна, это разовая задача на будущее и не блокирует
закрытие тикета.

**Суть изменений упрощенно.** Раньше, если на сервере что-то ломалось,
единственным способом узнать об этом было вручную зайти в Railway и читать
сырые текстовые логи — без всякой гарантии, что кто-то вовремя туда
заглянет. Теперь у приложения появилась своя «нервная система»:

1. У каждого запроса к серверу теперь есть уникальный номер, который виден
   и клиенту, и во всех связанных с ним строках лога — можно за секунды
   найти именно то, что произошло с конкретным запросом конкретного
   пользователя, а не листать тысячи безликих строк в поисках нужной.
2. Любая непредвиденная ошибка сервера теперь автоматически и подробно
   записывается в аккуратном машиночитаемом формате (JSON), а не свободным
   текстом — это значит, что логи можно в будущем без переписывания кода
   подключить к любой системе анализа или фильтрации.
3. Появилась отдельная проверка «жив ли Redis-кэш», а не только «жива ли
   база данных» — раньше эта часть системы была невидима для мониторинга.
4. Раз в 2 часа, когда и так уже запускается сборщик данных (сидер),
   система заодно тихо проверяет здоровье БД и Redis — и если что-то не в
   порядке, сама отправляет письмо на почту, вместо того чтобы полагаться
   на то, что проблему случайно заметят в логах.

Итог: проект теперь умеет сам сигнализировать «у меня проблема», а не молча
ждать, пока это заметит человек. Осознанно не стали подключать платные
внешние сервисы вроде Sentry — сделали урезанную, но полностью бесплатную
версию той же идеи своими силами (см. компромиссы выше).

**Верификация в проде (2026-07-04, ручной запуск сидера владельцем):**
прогон `workflow_dispatch` (run `28717613226`, 12m45s) подтвердил обе
новые фичи на живом Railway — GC: `Удалено статей-сирот: 0` (в этом цикле
сирот не появилось), health-check: `Всё в порядке` (письмо не
отправлялось — деградации не было). Адрес для алертов (`FROM_EMAIL`)
проверен владельцем — корректный.

**Закрыто:** issue #48.

**Известный пробел → кандидат в следующий тикет.** `X-Request-ID` сейчас
живёт только в HTTP-заголовке ответа — фронтенд нигде не показывает его
пользователю при ошибке (нет ни блока «Error ID: xyz», ни его копирования
в error-toast). Механизм корреляции запрос→лог пока полезен только самому
владельцу при отладке через DevTools/Railway, но не работает как канал
обратной связи от реальных пользователей о конкретных ошибках. Следующий
тикет: фронтенд ловит 5xx-ответы и показывает пользователю короткий
error-id (из `X-Request-ID`) с просьбой прислать его в поддержку — это
замыкает петлю «юзер увидел ошибку → написал → нашли конкретный traceback
по id за секунды». Без этого наблюдаемость остаётся односторонней
(сервер→владелец), а не (пользователь→владелец).

### Нагрузочное тестирование и автоматизация A11y (План действий)

По итогам ревизии текущей кодовой базы (июль 2026), инфраструктура проекта уже обладает высокой степенью зрелости:
- Бэкенд-тесты централизованы в `/tests` (разделены на `unit` и `integration`) и используют `pytest`.
- Фронтенд-тесты (`vitest`) используют co-location в `frontend/src` и интегрированы в жесткий CI-пайплайн (`frontend-tests.yml`).
- Настроен линтинг как для фронтенда (`eslint` в flat-формате `eslint.config.js`), так и для бэкенда (`ruff`, `mypy`).

Ниже представлено оптимизированное Техническое Задание (ТЗ) на закрытие пунктов 4 и 5 из обратной связи, использующее исключительно уже внедренные паттерны.

#### Шаг 1. Сдвиг проверок A11y влево (Shift-Left)
**Цель:** Автоматизировать A11y-проверки без создания новых CI-джобов.
**Действия:**
1. **Установка:** `npm i -D eslint-plugin-jsx-a11y vitest-axe`.
2. **Линтинг (статика):** Интегрировать `eslint-plugin-jsx-a11y` в `frontend/eslint.config.js` (с учетом нового flat-config формата). Это мгновенно добавит A11y-проверки в существующий `npm run lint`.
3. **Тесты (динамика):** Расширить `frontend/src/test/setup.ts` матчерами `vitest-axe`. Написать 1-2 тестовых кейса для критичного UI (например, `TableBuilder`), используя утверждение `expect(await axe(container)).toHaveNoViolations()`, что позволит ловить ошибки рендера в джобе `unit`.

#### Шаг 2. Baseline нагрузочного тестирования
**Цель:** Создать воспроизводимую историю производительности (baseline) для демонстрации работодателю.
**Действия:**
1. **Инструментарий:** Использовать `k6` (наиболее современный инструмент, не требующий тяжелого Python-окружения вроде Locust). Скрипт разместить в `tests/load/baseline.js`.
2. **Сценарий:** Написать скрипт, симулирующий базовую нагрузку на 2-3 критичных read-only эндпоинта (полнотекстовый поиск `/api/articles` и выдача графиков `/api/stats/journal-landscape`).
3. **Документирование:** Запустить скрипт локально против сидированной БД (122k записей). Извлечь показатели P95, P99 и RPS, после чего описать результаты (включая команду запуска) в секции `Engineering decisions / Lessons learned` в `README.md`.

### Нагрузочное тестирование и автоматизация A11y — реализация, сбой и восстановление (2026-07-08 — 2026-07-09)

**Первая попытка (Agy CLI, ветка `feat/a11y-load-test`, 2026-07-08).** ТЗ из раздела выше реализовано
почти дословно за 5 коммитов (`c4c04f7` → `5ab811b`): триггеры CI на ветку, `eslint-plugin-jsx-a11y` +
`vitest-axe`, фикс найденных a11y-нарушений (`ChartCard`, `PivotTable.SortableHeader`, `pagination.tsx`,
`input-group.tsx`), скрипт `tests/load/baseline.js` на k6 + числа в README, финальный коммит с попыткой
починить TS-ошибку типов `vitest-axe`. По итогам последнего коммита **все** фронтенд CI-джобы и деплой
Vercel красные.

**Критический разбор подхода (Claude, 2026-07-09, до вмешательства в код).** Категории инструментов
выбраны верно (статика + рантайм-a11y, k6 без тяжёлого окружения, read-only эндпоинты без JWT/квоты
Scopus для нагрузочного теста), решение не заводить отдельную CI-джобу — соответствует уже сложившемуся
в проекте паттерну piggyback (GC/health-check на seeder-cron и т.п.). Но по существу — два независимых

### Индексирование БД под нагрузку — ТЗ (2026-07-09)

Root cause (прогон выше): оба эндпоинта делают parallel seq scan по 142k строк — ни один
индекс не соответствует форме запроса. План — по стандартам 2026 для read-heavy OLTP:
sargable-предикаты вместо функциональных индексов где возможно, `CONCURRENTLY` на живой
таблице, measure-first до/после через `EXPLAIN (ANALYZE, BUFFERS)`.

#### Шаг 1. Поиск (`title`/`author` ILIKE) — `pg_trgm` GIN, не `tsvector`
`tsvector` меняет семантику (word-boundary + стемминг вместо substring-match) — сломает
текущий UX «искать по подстроке в любом месте». `pg_trgm` сохраняет её 1:1, просто даёт
индекс под уже существующий паттерн запроса.
1. `CREATE EXTENSION IF NOT EXISTS pg_trgm;` — в Supabase доступно out-of-box, сверить
   `list_extensions` перед миграцией.
2. Два отдельных `GIN`-индекса, не один составной (фильтр через `OR`, планировщик сам
   сделает `BitmapOr` по двум индексам): `USING gin (title gin_trgm_ops)` и
   `USING gin (author gin_trgm_ops)`.
3. `GIN`, не `GiST` — read-heavy профиль (частые SELECT, bulk-write от сидера раз в 2ч):
   GIN быстрее на чтении ценой более медленной сборки — здесь верный компромисс.
4. `CONCURRENTLY` обязателен (иначе build блокирует таблицу на запись на проде) —
   Alembic-миграция вне транзакции (`op.get_context().autocommit_block()`).

#### Шаг 2. Year-фильтр — sargable-предикат, не функциональный индекс
Не индексировать `extract(year FROM publication_date)` — переписать сам запрос в
sargable-диапазон: `publication_date < make_date(max_year + 1, 1, 1)` + обычный
`CREATE INDEX CONCURRENTLY` на `publication_date` без выражения. Планировщик получает
Index Range Scan вместо seq scan бесплатно, без индекса под конкретный вызов `EXTRACT`.

#### Шаг 3. Верификация (обязательна перед закрытием тикета)
1. `EXPLAIN (ANALYZE, BUFFERS)` до/после на том же изолированном прод-масштабном снапшоте.
2. Повторный `tests/load/baseline.js` (20 VU, та же БД) — apples-to-apples с уже
   опубликованным `P95=11.89s`/`P99=13.39s`.
3. `pg_stat_user_indexes` через 1-2 недели прод-трафика — подтвердить реальное
   использование новых индексов, не write-only bloat, прежде чем считать закрытым.
