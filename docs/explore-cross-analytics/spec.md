## 0. Контекст и входные данные

Пользователь запросил 3 новых **стационарных** графика под текущим рядом из 6 KPI-тайлов (`KpiRow`/`DimensionDrawer`), представляющих **комбинированные** разрезы данных (в отличие от KPI/drawer, которые всегда одномерны):

1. **Top Countries by Year** — full width, линейный график как текущий `Publications by Year`, но топ-10 стран отдельными линиями.
2. **Sunburst** (2–3 уровня) — левая половина второй строки. Уровень 1 задан пользователем (топ-5 стран); уровни 2–3 и их порядок — предмет анализа этого документа.
3. **Top Journals × Country** — правая половина второй строки, вертикальные stacked-бары: топ-10 журналов, сегменты — страны авторов.

Пункт 4 запроса (список других ценных комбинаций для будущего Chart Builder) — вынесен в **Приложение A**, не реализуется в этой ветке.

**Решённые вопросы (второй раунд диалога):**
- Sunburst-порядок: **Country → DocType → OpenAccess**, подтверждено.
- Каждый сегмент каждого уровня — интерактивный (glow-рамка при наведении + tooltip с подписью и значением), цвета — контрастны в обеих темах.
- Цвета топ-5 (sunburst/график 3) — **подмножество той же палитры**, что топ-10 (график 1), с оговоркой на контрастность (допустимо отличие по светлоте/насыщенности, не по оттенку/hue).
- «Other»-сегмент в sunburst — оставить видимым, даже если статистически малозначим.
- `cited_by_count` — отложено до Chart Builder (см. Приложение A, п. 3/4/6).

## 2. Backend — новые агрегаты (обязательный скоуп, ранее не обсуждался)

Все 3 графика требуют **многомерных** GROUP BY, которых `StatsResponse`/`get_stats()` сейчас не отдаёт (только одномерные `by_year`/`by_country`/`by_doc_type`/`by_journal`). Это не чисто фронтенд-задача.

### 2.1 Новые поля `StatsResponse` (`app/schemas/article_schemas.py`)

```python
class YearCountryCount(BaseModel):
    year: int
    country: str
    count: int

class SunburstSegment(BaseModel):
    country: str
    doc_type: str       # один из фиксированного топ-5 + "Other"
    open_access: bool
    count: int

class JournalCountryCount(BaseModel):
    journal: str
    country: str         # один из топ-5 + "Other"
    count: int
```

Добавить в `StatsResponse`:
```python
by_year_top_countries: List[YearCountryCount]
sunburst_country_doctype_oa: List[SunburstSegment]
top_journals_by_country: List[JournalCountryCount]
```

### 2.2 Реализация в `PostgresCatalogRepository.get_stats()`

Критично: **переиспользовать уже посчитанные `by_country`/`by_journal`/`by_doc_type`** (та же сортировка `count DESC`, тот же `catalog_articles_q`), а не пересчитывать топ-N отдельными запросами — иначе есть риск рассинхронизации топ-5/топ-10 между новыми графиками и остальным дашбордом (KPI, drawer).

- **Топ-10 стран по годам** (график 1): взять первые 10 label'ов уже вычисленного `by_country`, `WHERE affiliation_country IN (...)`, `GROUP BY year, affiliation_country`.
- **Sunburst** (график 2): топ-5 стран = первые 5 label'ов `by_country`. Топ-5 типов документа — **фиксированный глобальный набор**, не пересчитанный на страну: первые 5 label'ов уже вычисленного `by_doc_type` (без `LIMIT`, там сейчас полный список ~12 типов). Эмпирически подтверждено: глобальный топ-5 (`Article`/`Conference Paper`/`Review`/`Book Chapter`/`Note`) покрывает **99.3%** данных и совпадает по составу для всех топ-5 стран (порядок Article/Conference Paper внутри — единственное отличие, для India). Использование фиксированного набора вместо top-5-на-страну даёт единую цветовую легенду по всему графику. `CASE WHEN document_type IN (:top5) THEN document_type ELSE 'Other' END`, `GROUP BY country, doc_type_bucket, open_access`.
- **Топ-10 журналов × страна** (график 3): топ-10 label'ов `by_journal`. Страны бакетируются в те же топ-5 (та же переменная, что в sunburst) + `'Other'`. `CASE WHEN affiliation_country IN (:top5) THEN affiliation_country ELSE 'Other' END`.

Ни новых индексов, ни миграций не требуется — используются уже существующие функциональные индексы (`lower(affiliation_country)`/`lower(document_type)`, migration `0014`) и тот же `SET LOCAL work_mem='32MB'` guard (`dialect=="postgresql"`).

### 2.3 Redis-кэш — без изменений

Новые поля — часть того же `StatsResponse`/того же вызова `get_stats()` → один и тот же cache-key (`db_namespace` + фильтры), тот же TTL=60s. Кэш-инфраструктуру трогать не нужно.

## 3. Общая фронтенд-инфраструктура (реализовать до самих графиков)

### 3.1 Вынос year-range констант

`YEAR_HARD_MAX`/`YEAR_DEFAULT_MIN`/`YEAR_MIN_WINDOW` сейчас объявлены только внутри `DimensionDrawer.tsx`. График 1 использует тот же слайдер/диапазон → вынести в `frontend/src/constants/yearRange.ts`, импортировать в оба места. Без этого — риск рассинхронизации границ при будущих правках (уже было полезно консолидировать при прошлой правке в рамках одного файла — теперь то же самое между двумя файлами).

`zeroFillYears()` (`chartColors.ts`) сейчас работает с одной серией. Для графика 1 нужна **новая** утилита `pivotYearCountrySeries()` (не расширение существующей сигнатуры другим форматом — развели бы одну функцию на два несовместимых shape) — превращает `YearCountryCount[]` в wide-формат `{year, [country]: count, ...}[]`, который требуют Recharts `Line`/`Area` для мультисерийного графика, с zero-fill внутри выбранного диапазона слайдера для каждой страны независимо.

### 3.2 Цветовая система — `getCountryColor()`

Риск, который нужно закрыть явно: топ-N стран — не константа, а результат запроса к живым данным. Если ранжирование сдвинется (новый сидинг), **позиционная** привязка цвета (индекс в массиве → цвет палитры) заставит страны менять цвет между визитами — путает пользователя, сравнивающего дашборд во времени.

Решение: `constants/countryColors.ts` — `COUNTRY_HUE_MAP: Record<string, number>` с явно закреплённым hue примерно для 25–30 стран, типично доминирующих в Scopus-выборках (Китай/США/Индия/Великобритания/Южная Корея/Германия/Япония/Австралия/Канада/Франция/Италия/Испания/Бразилия/Иран/Саудовская Аравия и т.д.), плюс **fallback** — детерминированный hash строки названия страны → hue, для любой страны вне списка (защита от «страна выпала из палитры → белый/пустой сегмент»). `getCountryColor(country, theme)` возвращает готовый цвет с уже применённой theme-aware светлотой/насыщенностью (тот же принцип, что `getRankedBarColor`/`TAXONOMY_PALETTE` — контраст к фону своей темы, не выцветание в неё).

Топ-5 в sunburst/графике 3 — те же hue, что топ-10 в графике 1 (по требованию пользователя), но с собственной светлотой/насыщенностью под контекст (тонкие дуги sunburst vs широкие линии) — это и есть согласованная пользователем «оговорка на контрастность».

Уровень 2 sunburst (тип документа) — переиспользовать существующую `TAXONOMY_PALETTE`/`getTaxonomyColor()` (уже валидирована на контраст в обеих темах для donut `doc_type` в `DimensionDrawer`). Уровень 3 (open access) — переиспользовать существующие цвета `DIMENSION_COLORS.open_access` (уже используются в `DrawerOAChart`). Новых цветов для этих двух уровней не изобретать — экономия риска, обе палитры уже прошли визуальное QA в прошлой ветке.

### 3.3 Sunburst — техническое решение

**Recharts не имеет встроенного типа Sunburst.** Рассмотрены варианты:

| Вариант | Плюсы | Минусы |
|---|---|---|
| **3 вложенных Recharts `Pie` (разные `innerRadius`/`outerRadius`, общий `PieChart`)** | Zero новых зависимостей; переиспользует существующие `ChartTooltip`/тему/hover-паттерны | Нужна аккуратная подготовка данных |
| `@visx/hierarchy`+`@visx/shape` (d3-based) | Более явный API для radial partition | Новая зависимость, свой набор примитивов, не интегрирован с текущей тема-системой |
| `@nivo/sunburst` | Готовый компонент, быстрее всего | Тяжёлая зависимость (свой d3-стек и theming), рискует конфликтовать с `useTheme()`/`ChartTooltip`, дальше раздувает `vendor-charts` chunk (уже ~432kB gzip) |

**Решение: вариант 1**, без новой зависимости. Технически это стандартный приём: Recharts `Pie` авто-распределяет углы пропорционально `value` в пределах `startAngle`/`endAngle` (по умолчанию 0–360°). Если данные каждого следующего кольца **отсортированы в том же порядке родителя** и **сумма значений кольца равна сумме значений родительского кольца**, дуги автоматически выравниваются под родителя — ручная арифметика углов не нужна, только правильная подготовка/сортировка данных (`flatten` в порядке country → doc_type → open_access).

**Митигации рисков конкретно для sunburst:**
- *Тонкие дуги на 3 уровне* (напр. UK × Book × true = 1 статья из 119 950) — визуально неразличимы/некликабельны. Митигация: `applyMinimumArcFloor(data, minDegrees≈1.5°)` — сегменты ниже порога получают минимальный угол для рендера (компенсируется пропорциональным сжатием остальных), но tooltip/подпись показывают истинное значение, не отрисованное.
- *Touch-устройства*: hover glow не существует на тач-экранах → добавить `onClick` (не только `onMouseEnter`) как альтернативный триггер активного состояния/tooltip.
- *Glow-рамка*: активный `Cell` получает `stroke` = theme-target цвет (белый в dark / почти-чёрный в light — та же логика, что `getRankedBarColor`), увеличенный `strokeWidth`, плюс SVG `filter: drop-shadow(...)` для собственно свечения.
- *Округление углов между 3 уровнями* — возможен субпиксельный зазор на стыках колец (косметический, не блокер).

## 4. График 1 — Top Countries by Year

Full width, под `ActiveFilterBanner`. Технически — расширение уже готового пайплайна `Publications by Year` (тот же `YEAR_HARD_MAX=2030`/слайдер/`Slider` из shadcn), 10 линий вместо одной, данные — `pivotYearCountrySeries(stats.by_year_top_countries)`.

**Риск и митигация — визуальный шум/доминирование Китая:** Китай (43 943) на порядок больше следующей страны (США, 13 174) → на линейной шкале остальные 9 линий сожмутся в тонкую полосу у нуля. Решение: **кликабельная легенда** (Recharts `Legend` с `onClick` → toggle видимости серии) — стандартный паттерн для многосерийных графиков с большим разбросом величин, позволяет пользователю временно скрыть Китай и рассмотреть остальные страны. Логарифмическая шкала не подходит — `zeroFillYears` создаёт нули для лет без данных, `log(0)` не определён.

Цвета линий — `getCountryColor(country, theme)` (см. §3.2).

## 5. График 2 — Country → DocType → OpenAccess Sunburst

- **Уровень 1** (внутреннее кольцо) — топ-5 стран, цвет `getCountryColor()`.
- **Уровень 2** — топ-5 типов документа (глобальный фиксированный набор, см. §2.2) + «Other», цвет `getTaxonomyColor()`.
- **Уровень 3** (внешнее кольцо) — Open Access true/false, цвет `DIMENSION_COLORS.open_access`.

Интерактивность: hover/tap → glow-рамка активного сегмента (см. §3.3) + tooltip: подпись измерения (переведённая) + конкретное значение + count + % от родительского сегмента (не от общего итога — сохраняет семантику drill-down).

Оформление — частично заимствовать референс пользователя (Disk Usage Analyzer): контрастные насыщенные цвета по кольцам, тонкие разделительные зазоры между секторами, скругление внешнего края не требуется (референс — острые сектора).

## 6. График 3 — Top-10 Journals × Country (вертикальные stacked-бары)

Топ-10 журналов (`stats.by_journal[:10]`), каждый бар — stacked по топ-5 странам + «Other» (тот же набор стран, что sunburst — единая легенда по всему дашборду). Цвета сегментов — `getCountryColor()` (консистентно с графиками 1/2).

**Риск — длинные названия журналов:** (напр. «Proceedings of SPIE the International Society for Optical Engineering») на узкой правой половине второй строки. Митигация — переиспользовать уже существующий паттерн `labelMaxLen` (`DrawerBarChart`/`DrawerCountryChart`) с усечением + полное название в tooltip; угол подписи по оси X — переиспользовать -40°, уже подобранный для `DrawerCountryChart`.

Подтверждено эмпирически (топ-10 журналов × страна на реальных данных): разрез не вырождается в «везде Китай» — например, *Lecture Notes in Networks and Systems* лидирует Индией (545 из ~700), *IEEE Access* — довольно ровное распределение. Разрез содержательный.

## Приложение A — кандидаты для будущего Chart Builder (справочно, не реализуется в этой ветке)

Ранжировано по аналитической ценности/востребованности в индустриальных научных дашбордах:

1. **OA-доля по годам** (`% open access` по годам, 100%-stacked area) — динамика перехода к открытому доступу.
2. **OA-доля по странам** (rate, не абсолютное число) — раскрывает политику (напр. Plan S в Европе), ортогонально существующим объёмным графикам.
3. **Импакт по типу документа** — среднее/сумма `cited_by_count` по `document_type`. Оговорка: медиана цитирований по всей коллекции = 0 (сильный zero-inflated skew) → использовать среднее или total, не медиану.
4. **Импакт по странам** — среднее цитирование как proxy качества, не только объёма.
5. **Тренд типов документов по годам** (stacked area) — эволюция состава публикаций во времени.
6. **Journal landscape scatter** — топ-N журналов: X = число статей, Y = среднее цитирование (классический библиометрический квадрант «объём vs импакт», как в VOSviewer/SciVal).
7. **OA-доля по топ-10 журналам** — раскрывает fully-OA издания (Scientific Reports, Nature Communications) на фоне смешанных.

`author` как измерение для конструктора — не рекомендуется: поле — плоская строка без ORCID/ID, риск ложных агрегаций по однофамильцам (`"Wang D."` и т.п.).
